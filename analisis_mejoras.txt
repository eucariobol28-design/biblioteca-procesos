ANÁLISIS DE LA APLICACIÓN: Mejora de la Plataforma "Biblioteca - Procesos Técnicos"
Proyecto: biblioteca-procesos
Fecha: 2026-06-10
Autor: Equipo / Generado por asistente

1. Resumen funcional
- Propósito: Aplicación de escritorio local (Tkinter + SQLite) para gestionar los procesos técnicos de una red de bibliotecas: recepción física de libros, catalogación (fichas y cotas), cola de impresión de cotas, distribución a sedes satélite, inventario por sala/biblioteca, gestión de usuarios y catálogos (salas, géneros, bibliotecas), y reportes mensuales.
- Público objetivo: administradores y operadores de procesos técnicos de bibliotecas regionales.

2. Arquitectura y tecnologías
- Lenguaje: Python 3.x
- UI: tkinter (tkinter.ttk)
- BD: SQLite (archivo `biblioteca.db` local)
- Estructura: modular MVC-like (carpetas `core`, `database`, `models`, `controllers`, `views`)
- Componentes claves:
  - `database/migrations.py`, `database/conexion.py` — inicialización y esquema
  - `models/*` — acceso y lógica ligera por entidad (Libro, Inventario, Ficha, Distribucion, Genero, Sala, Biblioteca, Usuario)
  - `controllers/*` — orquestación por caso de uso (BibliotecaController es controlador central)
  - `views/*` — pantallas Tkinter por flujo
  - `main.py` — arranque y ruteo entre vistas

3. Flujo principal (resumen)
- Login -> Panel principal -> Recepción (registrar libro y stock en Sede Central) -> Catalogación (calcular/aprobar cota, crear ficha) -> Cola de impresión (previsualizar/imprimir cotas) -> Distribución (generar actas, mover stock entre sedes/salas) -> Reportes mensuales.

4. Observaciones sobre el estado actual (limitaciones / puntos detectados)
- Esquema de BD: ha sido ampliado para soportar `genero_id`, inventario por `biblioteca_id` y `sala_id`, pero convive con columnas antiguas en `libros` (campos legacy), lo cual requiere limpieza o migraciones inteligentes.
- Concurrency / transacciones: se usan transacciones manuales en algunos puntos; SQLite es local y no concurrente para múltiples procesos, por lo que hay riesgo si varios usuarios intentan acceder al mismo archivo simultáneamente (no provisto de servidor centralizado).
- Validaciones en UI: entradas mínimas realizadas, pero faltan validaciones más estrictas (ISBN formato, longitud, duplicados más claros, manejo de campos vacíos, control de cargas masivas).
- Manejo de errores: errores se muestran con messagebox, pero falta logging estructurado para depuración y trazabilidad (ficheros `logs/` y niveles).
- Tests: no hay pruebas automatizadas (unitarias o de integración).
- Seguridad: contraseñas se almacenan con SHA-256 (mejor que texto plano), pero no hay sal (salt) ni PBKDF2/Argon2 para proteger hashes. No hay gestión de sesiones o bloqueo de intentos.
- UI/UX: interfaz básica Tkinter sin guía contextual; no hay paginación ni búsqueda avanzada en tablas grandes.
- Internacionalización: textos en español fijos; no existe soporte i18n.
- Paquetes/Dependencias: proyecto usa solo librerías stdlib; esto facilita despliegue, pero falta definir un `requirements.txt` o `pyproject.toml` para reproducibilidad (aunque stdlib evita muchas dependencias externas).

6. Recomendaciones técnicas concretas
- Migraciones: crear un pequeño módulo `database/migration_versions` que guarde versión y aplique scripts incrementales (idempotentes). Evitar ALTER con constraints no permisibles en SQLite; usar técnicas de tabla temporal cuando sea necesario.
- Hashing: migrar a PBKDF2-HMAC-SHA256 con salt único por usuario y un número de iteraciones configurable (ejemplo con `hashlib.pbkdf2_hmac`). Opcional: usar `argon2-cffi` si se permite dependencia externa.
- Logging: usar `logging` con `RotatingFileHandler`; colocar archivos en `logs/app.log` y no en el repo.
- Tests: añadir `tests/` con pytest y crear tests que usen bases de datos temporales en memoria (`sqlite3.connect(':memory:')`).
- UI: añadir utilidades de confirmación y validadores compartidos; refactorizar vistas para desacoplar lógica de presentación (extract helpers en `core/view_engine.py`).
- Backups: política simple de backups del `biblioteca.db` (copia horaria o al cierre de sesión) y comando UI para exportar copia.
